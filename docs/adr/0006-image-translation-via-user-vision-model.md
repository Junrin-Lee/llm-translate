# Image Translation sends the image to the user's own vision-capable model, no bundled OCR

**English** · [简体中文](./0006-image-translation-via-user-vision-model.zh-CN.md)

> **Note (2026-07-15):** the feature was renamed **Screenshot Translation (截屏翻译)** after the smoke pass — see CONTEXT.md for the current vocabulary. This ADR keeps the original name as a historical record.

Image Translation exists because both existing features die wherever the content script cannot run or text is not selectable — the top user complaints being the built-in PDF viewer and text baked into images. The image (a user-dragged crop of the current tab, or a pasted/uploaded file) is sent directly to the user-configured LLM as a multimodal chat message, extending both hand-rolled adapters (OpenAI-compatible `image_url` content parts; Anthropic-compatible `image` source blocks). We rejected the bundled-OCR route (e.g. Tesseract.js extracting text locally, then reusing the text pipeline): OCR quality on CJK, small, or stylized text is poor and discards layout the model can read visually; an OCR engine adds megabytes to an extension whose content script is resident on every page (the same size sensitivity that motivated ADR-0003); and it would bolt a second text-recognition pipeline onto a product whose philosophy is to stay a thin client over the user's own API (ADR-0002, ADR-0003). The accepted cost: image input becomes a capability requirement on the user's model. We deliberately do not maintain a vision-capability model list — it would rot and misfire on proxy/custom model names, the same reason ADR-0003 refuses vendor knowledge. Instead the feature resolves its provider through the existing Feature Override chain, and a protocol error from a non-vision model is normalized into an actionable message that deep-links to the Image Translation override in settings.

Interaction is dual-path because the trigger scenarios split exactly along injectability: on pages where the content script runs, the entry freezes the tab via `captureVisibleTab`, the user drags a region over the frozen snapshot (the overlay never contaminates the capture), and the result streams in place as a Translation Card; on restricted pages (built-in PDF viewer, browser-internal pages — precisely the complained-about ones) no overlay can exist, so the popup entry captures the visible area and opens an extension page hosting the same crop-and-translate flow, which also accepts pasted/uploaded images. The crop component is shared; only its host differs.

## Consequences

- `ChatRequest` and both adapters become multimodal; SSE parsing and error normalization tests must cover image payloads and the "model lacks vision" error mapping (extends the ADR-0003 testing priority).
- Crops are re-encoded and downscaled (JPEG/WebP, longest edge ≈1500–2000 px) before sending, to stay inside per-image provider limits and keep token cost bounded; exact numbers are fixed at implementation time.
- First use shows a one-time notice that the capture is sent to the user's configured API endpoint; `docs/privacy-policy.md` gains an Image Translation section.
- `activeTab` joins `permissions` so capture still works on Firefox after optional host access is revoked (verify during implementation; on failure, fall back to Permission Onboarding).
- Phase 1 ships the two capture paths plus paste/upload, entries in the popup and the context menu; right-clicking an image element (`srcUrl` route: cross-origin fetch, lazy-load placeholders) and a keyboard shortcut are phase 2.
- No translation cache for images in phase 1: repeat rate is low and content-hashing screenshots costs more than it saves.
