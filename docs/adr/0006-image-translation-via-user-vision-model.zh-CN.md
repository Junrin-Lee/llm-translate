# 图片翻译把图片直接发给用户自带的视觉模型,不内置 OCR

[English](./0006-image-translation-via-user-vision-model.md) · **简体中文**

图片翻译的存在理由是:现有两个功能在 content script 无法注入或文字不可选中的地方全部失效 —— 用户抱怨最集中的是内置 PDF 阅读器和"文字长在图片里"。图片(用户在当前标签页上拖拽框选的裁剪,或粘贴/上传的文件)作为多模态 chat 消息直接发给用户配置的 LLM,为此扩展两个手写适配器(OpenAI 兼容的 `image_url` content part;Anthropic 兼容的 `image` source block)。我们否决了内置 OCR 路线(如用 Tesseract.js 本地提取文字后复用现有文本管线):OCR 对中日韩、小字号、艺术字的识别质量差,且丢弃了模型本可以从视觉上读懂的版面信息;OCR 引擎会给一个 content script 常驻所有页面的扩展增加数 MB 体积(与 ADR-0003 的体积敏感理由相同);它还会在"做用户自有 API 之上的薄客户端"(ADR-0002、ADR-0003)这一产品哲学上外挂第二套文字识别管线。接受的代价是:图片输入成为对用户模型的能力要求。我们刻意不维护"支持视觉的模型名单" —— 名单必然过时,还会误拦代理网关/自定义模型名,这与 ADR-0003 拒绝维护厂商知识是同一个理由。取而代之:该功能通过现有功能级覆盖链解析 Provider,非视觉模型返回的协议错误被归一化为可行动提示,并深链到设置中图片翻译的功能级覆盖。

交互采用双路径,因为触发场景恰好沿"可否注入"分裂:content script 可运行的页面,入口先用 `captureVisibleTab` 冻结画面,用户在冻结截图上拖拽框选(遮罩层永远不会污染截图),结果以译文卡片原地流式输出;受限页面(内置 PDF 阅读器、浏览器内部页面 —— 恰是被抱怨的那些)无法绘制遮罩,于是 popup 入口截取可见区域后打开扩展页,承载同一套"裁剪并翻译"流程,该页同时接受粘贴/上传的图片。裁剪组件共享,仅宿主不同。

## Consequences

- `ChatRequest` 与两个适配器变为多模态;SSE 解析与错误归一化测试必须覆盖图片载荷及"模型不支持视觉"的错误映射(延续 ADR-0003 的测试优先级)。
- 裁剪结果发送前重编码并降采样(JPEG/WebP,长边约 1500–2000 px),以满足各协议的单图上限并控制 token 成本;具体数值在实现时敲定。
- 首次使用展示一次性提示:截图将发送至用户配置的 API 端点;`docs/privacy-policy.md` 增补图片翻译章节。
- `permissions` 增加 `activeTab`,使 Firefox 撤销可选站点权限后截取仍可用(实现时验证;失败则回落到权限引导)。
- 一期交付两条截取路径 + 粘贴/上传,入口为 popup 与右键菜单;右键图片元素直翻(`srcUrl` 路线:跨域获取、懒加载占位图)与快捷键为二期。
- 一期不做图片翻译缓存:截图重复率低,对截图做内容哈希的成本高于收益。
